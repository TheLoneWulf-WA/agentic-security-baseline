Protocol rationale — why each rule exists

CLAUDE.md is the operating manual — what to do, exact strings, paths, thresholds. This is the companion: why each rule is there, what it prevents, and how the layers compose. Reach for it when you’re wondering "wait, why do we do that?" before changing or skipping a rule.

Section 01

Session Behavior

How Claude reads competing signals and how it talks about what it actually can and can’t do. The two foundational rules below shape everything else — the rest of the protocol assumes Claude is reasoning from documented decisions and stating its limits before bluffing.

Rule 1.1

Project CLAUDE.md supremacy over platform auto-suggestions

When a platform plugin or hook (Vercel plugin, framework skill, tool-validation hook, etc.) auto-injects a suggestion that conflicts with a deliberate, documented decision in the project’s CLAUDE.md, silently ignore it. Don’t surface it, don’t reason through it, don’t suggest the migration.

Scope. Architectural and tech-stack choices only. This rule does not override safety rules (Review Protocol, Dependency Gate, Push Routing, Merge Gate). Those apply uniformly regardless of project context.

Figure 1.1 · authority hierarchy — what outranks what
Why this exists

A plugin pattern-matching on pages/ can’t know why the project is on Next.js Pages Router and not App Router. The reason might be inherited codebase, migration cost, a third-party SDK that doesn’t support App Router yet, team familiarity, or a deadline. The project’s CLAUDE.md captures that context; the plugin’s auto-suggestion is a guess from a system that can’t see any of it.

The scope carve-out matters. Tech-stack opinions are user-by-user judgment calls. Safety rules (no force push to main, run the security review, gate the install) are not — they apply uniformly because the cost of skipping them is catastrophic and the cost of running them is small. A plugin shouldn’t override your stack choice, but your stack choice shouldn’t override a safety rule either.

Rule 1.2

State capability limits upfront

Before engaging with a request that may be outside actual capabilities, say what can and can’t be done first, then proceed with what can. Don’t bury the limit at the end. Don’t skip it.

Common limits worth surfacing immediately:

  • Can’t watch video — metadata and stills only.
  • Can’t hear audio.
  • Can’t render RAW image formats (.dng, .cr2, .arw) without conversion.
  • Can’t evaluate subjective "feel" without a concrete artifact (screenshot, example).
  • Can’t evaluate real-time behavior without a dev server plus a browser automation tool.
Why this exists

Bluffing is the failure mode. An AI guessing at work outside its capabilities produces confident-sounding answers that may be entirely fabricated. The user can’t correct for false confidence they don’t know is false. Surfacing the limit before working means the user knows what they’re actually getting.

Honesty before work, not during it. A mid-response "actually, I can’t do this part" is much worse than a one-line upfront "here’s what I can and can’t do, want to proceed?"

Section 02

Review Protocol

A layered review stack that runs before code reaches a PR or production. No single layer catches everything; each catches what the next can’t. The strict gate blocks; the reminder gate flags; /code-review runs on every PR; and review-until-clean ensures a fix doesn’t silently introduce its own bugs.

Figure 2.0 · defense in depth — what catches what, in what order
Rule 2.0 · out-of-band

Layer 0 — security-guidance plugin

The security-guidance@claude-plugins-official plugin (enable at user scope) is the earliest layer of the review stack and runs automatically — nothing to invoke. It fires three classes of check, each at a different lifecycle point:

  • Per-edit: deterministic pattern match on every Edit / Write / NotebookEdit — no model call, no token cost. Custom patterns at ~/.claude/security-patterns.json (global), <project>/.claude/security-patterns.json (project), and <project>/.claude/security-patterns.local.json (gitignored, personal overrides on shared repos). JSON, not YAML — the plugin doesn’t install PyYAML.
  • End-of-turn + on-commit: fresh-context model-backed reviews (a separate Claude call, not self-grading) over the turn’s diff and over commits / pushes Claude makes via its Bash tool. Commits run from the ! shell escape bypass this layer. Findings come back as in-session fix instructions.
  • Threat-model guidance: ~/.claude/claude-security-guidance.md (global) + <project>/.claude/claude-security-guidance.md (project) + <project>/.claude/claude-security-guidance.local.md (gitignored, personal overrides) feed the model-backed reviewer alongside its built-in checklist.

Default model: Claude Opus 4.7. Override via SECURITY_REVIEW_MODEL (end-of-turn) and SG_AGENTIC_MODEL (on-commit) in ~/.claude/settings.json if you want to bound per-PR token spend with a smaller model.

Canonical docs: https://code.claude.com/docs/en/security-guidance

Why this exists

Out-of-band means it doesn’t cost session context tax. Its rule files don’t load into your session; the plugin reads them itself. It catches the cheap, high-frequency stuff — secrets, unsafe DOM APIs, eval, exported components, the IOC patterns you teach it — while you’re writing, so the strict gate at push time only has to think about what slipped past.

It does NOT replace the gates below. Non-blocking, security-only, can miss things — it reduces what reaches the gates; it does not satisfy any of them. Strict Gate still blocks on HIGH; /code-review --comment still runs on every PR.

Plugin findings are not subject to the "ignore platform auto-suggestions" rule from Session Behavior — they’re security findings, not pattern-matched tech-stack suggestions. Treat them as real signal, not noise.

Rule 2.1 · blocking

Strict Gate — run /security-review automatically

Before any git push, if the diff touches a strict-gate surface below, run /security-review first. Don’t ask permission; just run it. Block on HIGH — report findings and wait for the user’s decision. Proceed-and-report on MEDIUM or LOW.

The gate is structured as a universal core plus per-ecosystem extensions. New ecosystems (Solana mobile, embedded firmware, etc.) get their own sub-section as work expands rather than forcing one flat list.

Universal · any project
  • Payment processing
    Money flows are zero-tolerance. A subtle bug here is a refund line item or a regulator letter, not a typo.
  • Auth / session / JWT / token logic
    Auth bugs are blast-radius bugs. A weakened check on one route can compromise every protected route.
  • Env vars, secrets, config files
    A secret in the wrong place — logs, errors, client bundle, a committed .env — is exfiltration. Cleanup after the fact rarely cleans up everywhere.
  • API endpoints accepting user input
    Every endpoint is an attack surface until proven otherwise. Inputs need validation, size caps, and shape checks at the boundary.
Web-specific additions
  • Database RLS policies · Supabase service role
    RLS is the only thing standing between one user and another’s data. A policy change without review is a class-wide breach risk.
  • CORS configuration · allowed origins
    A wildcard or accidental subdomain inclusion opens the API to anyone’s frontend.
  • Redirect URLs · OAuth callbacks · payment callbacks
    Misconfigured redirects are how token theft happens. The callback URL is part of the trust boundary.
  • Next.js middleware · route protection
    Middleware runs before route handlers. A bypass here bypasses every downstream check.
  • Third-party scripts · iframes · remote embeds
    External code runs with your origin’s permissions. CSP and integrity matter here.
Android-specific additions
  • User-media decode paths
    BitmapFactory, MediaCodec, ContentResolver/SAF
    Malformed-input class: crashes, bitmap OOM, MIME confusion. Permissions are the door; capture code is the room.
  • JNI / native-lib boundaries
    Memory-corruption surface. Buffer-length and ownership errors crossing the JNI boundary are full-process compromise.
  • android:allowBackup, dataExtractionRules
    Controls OS-level data exfiltration. One wrong attribute and .env-equivalent state is backed up to Google Drive.
  • Exported components
    android:exported="true"
    Especially with <data> filters — any other app on the device can call in. Validate every inbound extra and URI.
  • WebView / JS↔Kotlin bridges
    addJavascriptInterface
    XSS in WebView plus a JS-to-native escape is full app compromise. Custom WebViewClient/WebChromeClient overrides touching data flows count.
  • External text → LLM prompt
    Any string from disk, network, share intent, deep link, or user input that lands in an LLM context is prompt-injection surface.
  • Network Security Config / usesCleartextTraffic
    Credential exposure. Prefer scoped exception domains over a global allow.
  • Crypto / KeyStore
    Authenticated encryption choice (GCM vs bare CBC), IV choice, key generation/storage. Hand-rolled crypto is almost always wrong.
iOS-specific additions
  • User-media decode
    AVCaptureSession, UIImagePickerController, PHPickerViewController, CGImageSource, AVAudioRecorder
    Same malformed-input / OOM / format-confusion class as Android. Decode defensively, cap size before allocation.
  • Native ↔ JS bridges
    WKScriptMessageHandler, JSContext/JSExport, RN/Expo native modules
    Analog of Android’s addJavascriptInterface. Validate everything crossing in.
  • Backup / iCloud exclusion
    NSFileProtectionComplete, excludedFromBackup, CloudKit config
    Wrong attribute backs sensitive data to iCloud where it’s out of your control.
  • URL schemes / Universal Links / extensions
    application(_:open:), continue userActivity
    Inbound URLs and share-extension NSItemProvider items are classic external attack surfaces.
  • Entitlements (*.entitlements)
    New entitlements grant capabilities (keychain group, app group, push, background modes) that can become exfiltration surfaces.
  • App Transport Security exceptions
    NSAppTransportSecurity, NSAllowsArbitraryLoads
    iOS equivalent of Android cleartext config. Prefer scoped exception domains.
  • Crypto / Keychain
    SecKey*, CryptoKit, CommonCrypto, Secure Enclave
    Same authenticated-encryption + IV + key-storage rules as Android.
  • External text → LLM prompt (on-device or cloud)
    Core ML, MLX, or cloud — any string from disk, URL scheme, NSItemProvider, share extension or user input injected into an LLM context is prompt-injection surface.
Why this exists

Sensitive surfaces have asymmetric blast radius. A subtle slip in auth, payments, secrets, exported components, or capture paths becomes a class-wide breach, not a single bug. The cost of a 2-minute automated review is much less than the cost of the bug, every time.

The universal-core-plus-extension structure scales without rewrites. New ecosystems (iOS shipping in 2026, Solana mobile shortly after) add their own sub-section instead of forcing one giant flat list that’s 80% irrelevant per project.

Blocking on HIGH only — not on MED or LOW — because the gate has to be useful, not noisy. MED/LOW report-and-proceed means the user sees the finding without being stopped for things that are usually safe. Push-gate caching (don’t re-run on the same branch unless new commits touch a new strict surface) keeps the cost reasonable on iterative push cycles.

Rule 2.2 · flag, don’t auto-run

Reminder Gate

For borderline surfaces, remind the user to consider running /security-review before pushing. Don’t auto-run.

Universal
  • File uploads · media handling
    Often safe; sometimes a path-traversal or upload-as-script vector.
  • Route changes · deep links
    New external surfaces appear silently; worth a beat.
  • Native bridges · webview communication
    Bridge changes can re-shape the trust boundary.
  • Database schema changes
    Mostly mechanical, occasionally policy-affecting.
Android-specific additions
  • targetSdkVersion bumps
    Shifts defaults — scoped storage, default-allowBackup, default-exported, runtime permissions. Policy change worth attention.
  • Release signing / ProGuard / R8 config
    Misconfigured rules can leak debug code into release or skip obfuscation on sensitive classes.
  • Plain SharedPreferences for sensitive data
    Should be EncryptedSharedPreferences; context-dependent, nudge rather than block.
iOS-specific additions
  • Deployment target bumps
    Changes default privacy UX, scoped storage rules, API behavior. Policy shift.
  • Release-build hardening config
    Strip Debug Symbols, optimization level, dead-code stripping. Misconfigured builds leak debug paths.
  • Plain UserDefaults for sensitive data
    Should be Keychain Services; nudge rather than block.
Why this exists
Some surfaces are worth thinking about but not always a hard block. Running /security-review on every schema bump or SDK bump is overkill and trains the user to mute the review. Nudging once, then trusting the user’s judgment, keeps the signal high.
Rule 2.3 · user override

Skip Clauses

If the user says "skip review", "just push", or "no review needed", bypass the strict gate for that one push. Don’t ask for confirmation. Trust the override.

Why this exists
Override should be friction-free when the user knows what they’re doing. Asking "are you sure?" after the user said "skip review" is anti-helpful — it trains the user to ignore the prompt, which weakens every future prompt. The skip is per-push, not per-session, so the cost of being wrong is bounded.
Rule 2.4 · on every PR

Always — /code-review --comment on every PR

Before requesting human review on any PR, run /code-review --comment.

In addition to what /code-review already checks, also flag:

  • any types, @ts-ignore, loosened TypeScript strictness
  • Empty catch blocks, swallowed errors, generic error handling that hides failures
  • console.log, commented-out code, TODO hacks, hardcoded test values
  • Performance issues: unoptimized queries, missing indexes, client-side fetching that should be server-side, unnecessary re-renders
  • Violations of project-level CLAUDE.md conventions
Why this exists
Free signal vs. requiring teammates to flag console.logs by hand. The point of automated review is to let humans review what only humans can — product judgment, architectural fit, "does this actually solve the problem" — instead of nitpicking style. Posting as a PR comment makes the findings durable so the next session can read them.
Rule 2.5 · mandatory loop

Review-until-clean — do NOT assume a fix worked

Applies to /security-review, /code-review, and any ad-hoc review-agent invocation. After applying a fix, re-run the review on the updated code. A passing TypeScript check is not a verified fix.

The loop:

  1. Run the review
  2. Apply fixes for any findings
  3. Commit + push the fixes
  4. Re-run the same review on the new state
  5. If new findings — repeat from step 2
  6. Stop only when a pass returns "no issues found"
Figure 2.5 · review-until-clean loop — the only exit is a clean pass
Why this exists

The failure mode is stopping after pass 1 because the fix "looks obvious." It often isn’t. A fix can introduce its own bugs. The reviewer’s fresh context on the updated code surfaces things the original pass missed, including issues in related files the fix touched implicitly.

For security findings the bar is higher: checking the severity-review label after the fix confirms the finding was downgraded or removed, not just reworded. A re-run that comes back with the same finding at the same severity is not a fix.

The override is the user’s explicit "ship it" / "good enough", not Claude’s judgment.

Carve-out for CLAUDE.md itself

Review findings on CLAUDE.md (this file, or any project-level CLAUDE.md) are not auto-fixable. The file is the rulebook; each finding is a protocol amendment, not a code fix. Discuss, decide, then apply or reject.

Mechanically re-running review-until-clean on the rulebook is the wrong loop — it can silently reword obligations and drift the protocol without deliberation. When /code-review fires on a CLAUDE.md change, treat the findings as proposals for amendment, not as defects to fix.

Rule 2.6 · reporting

Review output format

Inline: short summary, one screen max — finding count by severity, top 3 issues, recommended action.

Full report: saved to ~/.claude/reports/<project-name>/security-review-<branch>-<date>.md for later inspection. Same pattern for code-review output.

Why this exists
Reduces visible noise in the session while persisting the detail. Inline summary is what the user needs to decide; the full report is what they need later when investigating something that landed.
Section 03

Push Routing

How an unqualified "push this" gets routed. The default is strict — feature branch + PR + review + wait for ship-it — uniformly across every project, with explicit overrides for the hotfix case. There is no project class that gets less protection.

Rule 3.0 · default flow

Unqualified push → feature branch + PR + wait

When the user says "push this", "push it", "make a push", or any push command without specifying a target:

  1. Create a new feature branch from the current branch
  2. Push the branch to origin
  3. Open a PR via gh pr create
  4. Run /code-review --comment on the PR
  5. Stop and prompt the user to merge

The strict gate runs before the push, exactly as specified in Rule 2.1.

Figure 3.0 · default routing — the unqualified-push path
Why this exists

Default-safe. The most dangerous push is the unintended push to main. Defaulting to feature-branch + PR catches it — if the user really wanted to push to main, they’ll say so. Uniform protection across all projects means there’s no "personal project" category that gets less rigor (a "personal project" today can become a serious one tomorrow without the protocol catching up).

Rule 3.1 · user override

Explicit overrides (always honored)

  • "push to main" / "push to master" → push directly to main, no PR. The hotfix escape hatch.
  • "push to <branch-name>" → push to that branch, no PR creation unless requested.
  • "skip review" / "just push" / "no review needed" → bypass the strict gate (per Rule 2.3).
Why this exists
Explicit user intent should be honored without friction. The override is the user’s responsibility. The protocol is opinionated by default and unopinionated when told to be.
Rule 3.2 · convention

Branch naming default

Default branch name: claude/<short-slug-from-commit-message>.

Why this exists
Self-documenting — clear at a glance which branches were AI-initiated. Keeps the user’s normal feat/ namespace clean. Useful when scanning git branch -a a month later trying to remember what you actually wrote vs. what Claude generated.
Rule 3.3 · environment-aware

Graceful degradation

  • No origin remote → push to local branch only, tell the user the situation.
  • No gh CLI / unauthenticated → push to origin but skip PR creation. Tell the user so they can open the PR manually.
  • Never silently install tools or modify git config to make this work.
Why this exists
Silently fixing environment gaps hides what’s actually happening on the user’s machine. A side effect today is a "why is my git config different?" mystery tomorrow. Transparent degradation — do what you can, tell the user what you couldn’t — preserves the user’s mental model.
Section 04

PR Merge Follow-up

Opening a PR and merging it are separate decisions. The merge gate enforces that separation both at the instruction layer and at the hook layer — because instructions alone are best-effort.

Rule 4.0 · default behavior

Do not auto-merge — stop and prompt

After a PR is created and local reviews complete, do not auto-merge. Stop and tell the user: 

PR opened at <url>. Local checks passed (<summary>).
Say 'merge it' / 'ship it' / 'merge the PR' when you're ready.

Run gh pr merge --squash --delete-branch on the most recently opened PR when the user says "merge it" / "ship it" / "merge the PR" / "merge that".

Strategy overrides:

  • "merge as rebase" → --rebase
  • "merge as merge commit" → --merge
  • "merge but keep the branch" → omit --delete-branch

If multiple PRs are open in the session, ask which one to merge.

Why this exists
PR creation and merge are separate decisions. Auto-merging skips the window where you (or a teammate) would actually read the PR. The squash + delete-branch default keeps history clean and avoids stale-branch sprawl.
Rule 4.1 · mechanical backstop

Hook-enforced merge gate

~/.claude/hooks/push-routing-gate.sh (registered as a PreToolUse hook on Bash in ~/.claude/settings.json) catches every gh pr merge invocation and prompts for user confirmation.

Bypass phrase: "ship it through" (literal). When the user says exactly this phrase for a specific merge, Claude prefixes the command with the env-var bypass: 

MERGE_GATE_BYPASS=1 gh pr merge <pr> --squash --delete-branch

Any other phrasing — "ship it", "merge it", "merge the PR", "send it through", "just ship that", or any paraphrase — does not add the prefix. The bypass is per-merge, not per-session: each subsequent merge needs its own explicit phrase or its own confirmation.

Figure 4.1 · bypass mechanic — the literal phrase is the only key
Why this exists

Instructions are best-effort. Even with the "stop-and-wait-for-ship-it" rule in CLAUDE.md, a Claude in a high-momentum flow can drift past it and auto-merge. The hook is a mechanical backstop that doesn’t depend on the model choosing to follow the rule.

Why the bypass is a literal phrase, not a paraphrase set: the prefix has to be Claude’s explicit decision, not an inference from "they said ship-related words." A literal trigger means Claude can’t accidentally bypass by reading too much into normal merge phrasing. Per-merge (not per-session) bypass means defaults stay safe; each skip is re-affirmed.

Section 05

Dependency Installation Gate

Eight layers, from preventive to forensic. The first six are about choosing well; the last two are the structural defenses that closed the install-time execution gap exposed by the 2025 Axios attack and the 2026 TanStack / Mini Shai-Hulud campaign.

Figure 5.0 · dependency gate layers — preventive (cream) come first; forensic (terracotta) runs after the payload already executed
Rule 5.1 · vetting

Check the package first

Before npm install / npm add / npx <new-package> / pip install / equivalent:

  • Look up weekly downloads
  • Check first publish date (not last update date)
  • Check for known security advisories
  • Check if the package is deprecated or archived on GitHub
Why this exists
Most bad packages are detectable with simple signals. First-publish date matters more than last-update date because recently-created packages with high downloads are the typosquat pattern. A package "updated yesterday" that existed for five years is much safer than one that appeared yesterday with the same name as a popular library.
Rule 5.2 · veto signals

Flag and stop if any are true

  • Weekly downloads < 1,000
  • First published less than 30 days ago
  • Package name similar to a popular package (typosquat risk)
  • Ownership recently transferred
  • Package has postinstall, preinstall, or prepare scripts
Why this exists
Each signal individually is suggestive; combined they’re disqualifying. The script-presence flag is the most important one — it’s the install-time execution vector that Rule 5.7 closes globally, but a per-package heads-up still matters when overriding ignore-scripts for a specific install.
Rule 5.3 · user in loop

Report findings before proceeding

Don’t install without the user’s confirmation when findings come back.

Why this exists
Ambiguous cases are where the user’s judgment matters most. A 900-weekly-downloads package might be a legit niche tool or a typosquat; only the user knows which.
Rule 5.4 · post-install

After any dependency change

  • Run npm audit (or equivalent) and report results
  • Flag unexpected changes in package-lock.json: packages not explicitly requested, changed integrity hashes
Why this exists
Audit catches known vulns; the lockfile diff catches transitive surprises (a package you didn’t install showing up because a direct dep pulled it in).
Rule 5.5 · reproducibility

Version pinning

Always install exact versions (1.14.0), never ranges (^1.14.0 or ~1.14.0), unless the user says otherwise. If ~/.npmrc has save-exact=true, this is automatic.

Why this exists
Range specifiers silently pull in patches between machines and over time. Pinning to exact makes reproducibility the default and lets you deliberately bump when you want to, not silently when a patch happens to land.
Rule 5.6 · forensic layer

Lockfile version drift

When running npm install / npm update / npm ci, compare the lockfile before and after.

  • Flag any already-installed package whose version changed
  • Report which packages changed and what version they moved to
Why this exists — and what it does NOT do

This is a forensic layer, not a preventive one. By the time the lockfile diff appears, any malicious postinstall script has already executed. It tells you what changed; it does not prevent the change.

The preventive layers for trusted-package hijacking — the attack class that hit Axios in 2025, then the @tanstack/ namespace plus @opensearch-project/, @mistralai/, guardrails-ai, @uipath/, and squawk in the 2026 Mini Shai-Hulud campaign — are Rules 5.7 and 5.8 below.

Still: don’t silently accept unexpected version changes. Investigate.

Rule 5.7 · structural defense

Disable install scripts by default

Most supply chain attacks deliver their payload via preinstall, postinstall, or prepare scripts that run automatically during npm install. Disable script execution globally: 

npm config set ignore-scripts true

This eliminates the install-time execution vector. Compromised packages become drastically less dangerous because their payload can’t auto-run on install.

Trade-off: packages that legitimately need install scripts will fail on first install. Common cases (sharp, esbuild, puppeteer, cypress, husky, node-gyp, native modules) are listed in docs/CUSTOMIZE.md. Override per-install with npm install <pkg> --ignore-scripts=false or run npm rebuild <pkg> after a normal install.

Why this exists

The lockfile-drift check (Rule 5.6) is forensic — it runs after the payload already executed. The preventive fix is to remove the execution surface entirely. The trade-off is real (per-install overrides for legit cases), but the asymmetry is correct: a five-second annoyance during a deliberate sharp install vs. a worm running on every install.

This is the layer that would have stopped the 2026 Mini Shai-Hulud campaign at the install boundary on machines where it was set. The payload couldn’t auto-execute, so the persistence daemon never got planted.

Rule 5.8 · structural defense

Minimum release age

Require packages to be at least N days old before npm will install them:

npm config set min-release-age 2d

This shifts you out of the early-installer window for every future supply chain attack. Security researchers typically detect malicious uploads within 6–24 hours; a 2-day buffer means you skip the exposure window by default, without needing to know in advance which package will be hit next.

Requires npm 11+. Equivalents for pnpm, yarn, and bun are documented in docs/CUSTOMIZE.md. Older npm versions silently ignore the setting — upgrade to get the protection.

Trade-off: you’re 2 days behind on legitimate updates. Override per-install when a genuine same-day release is needed. This applies to direct installs and any package added as part of a build task. No exceptions.

Why this exists

Rule 5.7 kills the install-time execution vector. Rule 5.8 shrinks the window where you’d even pull in a compromised version. The detection community is consistently fast (Socket flagged the TanStack worm within hours of upload) — this rule converts "the community has noticed" into "I’m protected" without having to track every advisory by hand.

The two work together. 5.7 is "if I do install the bad package, the payload can’t run." 5.8 is "I probably won’t install it in the first place." Either alone is partial; both together close the gap that drift-detection alone couldn’t.

Section 06

Media Pre-Commit Gate

Git is the source of truth for code, not the storage for large binaries. Catch oversize media before staging, never commit source masters.

Rule 6.0 · pre-commit

Surface oversize, gitignore raw sources

  • Hard rule: surface any file > 5 MB to the user before staging. GitHub warns at 50 MB and rejects at 100 MB; catch oversize before push, not after.
  • Always gitignore raw/uncompressed sources: *.m4v, *.mov, *.dng, *.cr2, *.arw, *.nef, *.wav. Compressed derivatives only in the repo.
  • Project-specific compression targets (JPEG quality, CRF values, resolution ceilings) belong in the project's CLAUDE.md, not here.
Why this exists

Git history is permanent — large binaries bloat history forever. Once committed, removing them later requires destructive history rewrites that break every cloned copy of the repo. Compression at commit time is free; cleanup after the fact is expensive.

GitHub’s 100 MB rejection mid-push is a particularly annoying failure mode — the push fails, you don’t notice until you’ve already committed, and the fix requires rewriting commits. Prevent the class of problem at staging time.

Section 07

Global Pre-Push Git Hook

Defense against direct terminal pushes that bypass Claude entirely. Configured via core.hooksPath so it works regardless of repo-local hook setup.

Rule 7.0 · defense in depth

Global pre-push hook on protected branches

The pre-push hook should:

  • Only trigger on pushes to main, master, or production
  • Run npm audit --audit-level=critical if package.json exists, and block if critical vulnerabilities are found
  • Check if the diff touches sensitive files (auth, payments, env, middleware, RLS policies) and warn before allowing the push
  • Yield gracefully to project-local hook systems (like Husky) so it doesn't break repos that manage their own hooks
  • Be skippable with git push --no-verify for intentional bypass

If this hook is not yet installed, ask the user once per session whether to set it up. If yes, set it up. If no or skipped, don't ask again that session and don't configure git silently.

Why this exists

All the rules above are about pushes Claude makes through its Bash tool. A user running git push origin main from their terminal bypasses every one of them. The global git hook is the catch-all for that case — not blocking work, but adding a moment of attention before any push to a protected branch.

The yield-to-Husky rule matters: many projects have their own pre-push hooks for tests, linting, conventional commits. The global hook composes with those rather than competing with them. --no-verify remains the universal escape hatch for "I know what I’m doing, just push."

Maintenance discipline

This file mirrors CLAUDE.md. When you change a rule, update the corresponding section here in the same commit. The CHANGELOG.md entry is the audit trail — note the doc updated alongside the rule. Drift starts the moment one moves without the other.

The split is deliberate: CLAUDE.md gets loaded into every Claude Code session and is sized for context cost; this file is a reference fetched on demand. Resist the urge to merge them back together.
